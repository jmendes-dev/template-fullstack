# Internacionalização (i18n)

**Status: opcional.** Ler apenas se o projeto precisa suportar múltiplos idiomas. Projetos só em pt-BR não consomem este doc.

Princípio: i18n agregado desde o dia 0 é fácil; i18n retroativo é doloroso. Se houver qualquer chance de suportar outro idioma, preparar as chaves desde o começo.

## Quando adotar

| Cenário | Decisão |
|---|---|
| App interno do Brasil apenas | Não — strings hardcoded em pt-BR |
| Produto SaaS B2B com clientes potenciais fora BR | Sim — começar dia 0 |
| App em pt-BR com plano futuro "talvez en" | Sim — chaves desde o começo, traduzir quando precisar |
| App técnico cujo idioma é inglês (docs, UI técnica) | Inglês direto, sem i18n |

## Lib recomendada: `@lingui/core` + `@lingui/react`

Mais leve que `react-i18next`, integra com macro/extração automática. Alternativa: `react-i18next` (mais popular, curva mais simples, bundle maior).

Instalação (setup com Vite — sem Babel):

```bash
bun add @lingui/core @lingui/react
bun add -d @lingui/cli @lingui/vite-plugin
```

Adicionar plugin no `vite.config.ts` (**padrão recomendado — sem Babel**):

```typescript
import { lingui } from '@lingui/vite-plugin';
import react from '@vitejs/plugin-react';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [
    react(),   // sem opção babel — @vitejs/plugin-react v6 usa Oxc por padrão
    lingui(),
  ],
});
```

`@lingui/vite-plugin` compila strings via `lingui extract && lingui compile` — sem dependência de Babel. É o caminho alinhado com a stack (Oxc/Rolldown, sem `@babel/core`).

> **Alternativa com macros Babel** (se precisar de `<Trans>` compilado em build-time sem `lingui compile` explícito): instalar `@babel/core` como devDependency e passar `babel: { plugins: ['@lingui/babel-plugin-lingui-macro'] }` no `@vitejs/plugin-react`. Isso reabilita o pipeline Babel no plugin, adicionando ~50 MB de devDeps. Só adotar se o workflow de extração manual for inviável.

## Estrutura

```
apps/web/
├── src/
│   ├── locales/
│   │   ├── pt-BR/
│   │   │   └── messages.po
│   │   └── en/
│   │       └── messages.po
│   ├── i18n.ts                # setup Lingui
│   └── components/
├── lingui.config.ts
```

### `lingui.config.ts`

```typescript
export default {
  locales: ['pt-BR', 'en'],
  sourceLocale: 'pt-BR',
  catalogs: [
    {
      path: '<rootDir>/src/locales/{locale}/messages',
      include: ['<rootDir>/src'],
    },
  ],
  format: 'po',
};
```

### `src/i18n.ts`

```typescript
import { i18n } from '@lingui/core';
import { messages as ptBR } from './locales/pt-BR/messages';
import { messages as en } from './locales/en/messages';

i18n.load({
  'pt-BR': ptBR,
  en: en,
});

const stored = localStorage.getItem('locale') ?? navigator.language;
const locale = stored.startsWith('pt') ? 'pt-BR' : 'en';

i18n.activate(locale);

export { i18n };
```

### `main.tsx`

```tsx
import { I18nProvider } from '@lingui/react';
import { i18n } from './i18n';

createRoot(document.getElementById('root')!).render(
  <I18nProvider i18n={i18n}>
    <App />
  </I18nProvider>
);
```

## Uso nos componentes

### Strings estáticas

```tsx
import { Trans } from '@lingui/react/macro';

function Welcome() {
  return <h1><Trans>Bem-vindo!</Trans></h1>;
}
```

Macro extrai "Bem-vindo!" para `messages.po` no build.

### Strings dinâmicas com variáveis

```tsx
import { Trans } from '@lingui/react/macro';

function Greeting({ name }: { name: string }) {
  return <Trans>Olá, {name}!</Trans>;
}
```

### Plurais

```tsx
import { Plural } from '@lingui/react/macro';

<Plural
  value={count}
  one="# item"
  other="# itens"
/>
```

### Strings fora de JSX

```tsx
import { t } from '@lingui/core/macro';
import { i18n } from '@/i18n';

const title = t(i18n)`Configurações`;
toast.success(t(i18n)`Salvo com sucesso`);
```

## Mensagens de erro da API

API retorna códigos (`code: 'VALIDATION_ERROR'`), frontend traduz:

```typescript
// src/lib/api-errors.ts
import { t } from '@lingui/core/macro';
import { i18n } from '@/i18n';

const CODE_MESSAGES: Record<string, () => string> = {
  VALIDATION_ERROR: () => t(i18n)`Dados inválidos. Verifique os campos.`,
  UNAUTHORIZED: () => t(i18n)`Faça login para continuar.`,
  FORBIDDEN: () => t(i18n)`Você não tem permissão.`,
  NOT_FOUND: () => t(i18n)`Não encontrado.`,
  RATE_LIMITED: () => t(i18n)`Muitas tentativas. Aguarde um instante.`,
  INTERNAL_ERROR: () => t(i18n)`Erro interno. A equipe foi notificada.`,
};

export function translateApiError(code: string): string {
  return (CODE_MESSAGES[code] ?? CODE_MESSAGES.INTERNAL_ERROR)();
}
```

API não envia strings traduzidas (idioma do frontend ≠ do backend no geral).

## Extração e build

Ao desenvolver, rodar:

```bash
bunx lingui extract
```

Isso varre `src/`, encontra strings em `<Trans>`/`t`/`Plural` e atualiza `messages.po`.

Depois:

```bash
bunx lingui compile
```

Gera `messages.ts` consumível em runtime.

**Integrar no build**:

```json
{
  "scripts": {
    "dev": "lingui compile && vite",
    "build": "lingui extract && lingui compile && vite build"
  }
}
```

## Tradução

### Manual

Editar `.po` direto:

```
#: src/components/Welcome.tsx:5
msgid "Bem-vindo!"
msgstr "Welcome!"
```

### Editor visual

- [Poedit](https://poedit.net/) — free, offline
- [Crowdin](https://crowdin.com/) / [Lokalise](https://lokalise.com/) — SaaS com workflow de time, vale quando houver ≥3 idiomas ou tradutores externos

### LLM para primeira passada

Para draft inicial, usar LLM. Sempre revisar (LLMs erram em contexto cultural, formalidade, jargão específico).

## Seletor de idioma

```tsx
import { i18n } from '@/i18n';
import { useState } from 'react';

export function LocaleSwitcher() {
  const [locale, setLocale] = useState(i18n.locale);

  const change = (l: string) => {
    i18n.activate(l);
    localStorage.setItem('locale', l);
    setLocale(l);
  };

  return (
    <select value={locale} onChange={(e) => change(e.target.value)}>
      <option value="pt-BR">Português</option>
      <option value="en">English</option>
    </select>
  );
}
```

## Datas, números, moeda

**Nunca** string de locale. Usar `Intl` API nativa:

```typescript
new Intl.DateTimeFormat(i18n.locale).format(new Date());
new Intl.NumberFormat(i18n.locale, { style: 'currency', currency: 'BRL' }).format(1234.56);
new Intl.RelativeTimeFormat(i18n.locale).format(-1, 'day'); // "ontem" / "1 day ago"
```

Em pt-BR: formata como BR. Em en-US: formata como US. Zero lib.

## Anti-patterns

- Strings hardcoded misturadas com traduzidas — inconsistência visual
- Concatenar strings traduzidas com variáveis (`"Olá, " + name`) — gramática quebra em idiomas com flexão
- Traduzir mensagens do backend no backend — backend não conhece o idioma do frontend
- Locale via URL prefix (`/pt-BR/dashboard`) sem redirect inteligente — UX ruim
- Esquecer de extrair antes do build — strings novas não traduzidas
